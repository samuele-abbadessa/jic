# JIC CLI: Vendor & Submodules Support

**Date**: 2026-03-19
**Status**: Approved
**Approach**: Vendor as layer over ExecutionContext (Approach A)

## Problem Statement

JIC CLI currently assumes:
1. The working directory corresponds to a single project
2. There is no git repo at the root level — modules are independent git repos

A new project requires:
- Multiple git repos as subfolders (submodules of a root repo)
- Vendor-branch distribution model: different clients get different branches
- Coordinated checkout of vendor-specific branches across modules
- Vendor-specific configuration (env vars, AWS, K8s, branches)

## Design Decisions

| Decision | Choice |
|----------|--------|
| Config per vendor | Separate file in `.jic/vendors/jic.config.<vendor>.json` |
| Non-vendor modules on checkout | Checkout `master` (configurable per vendor) |
| Submodule pointer update | Only at specific workflow moments or `--update-root` |
| Sync conflict strategy | Fail-fast, re-run to continue |
| `jic vendor create` | Generates config file + branches (master, dev, build) |
| Session branch naming | `<vendor>/feature/<session_name>` in root + submodules |
| Vendor module group | Implicit `@<vendor>`, default for all commands |
| Session base branch | `<vendor>/dev` |
| Default vendor | "root" — always active, all modules on master, has its own config file |
| Backward compatibility | `type: "independent"` unchanged |
| Module outside vendor | Error (must `jic vendor add` or switch vendor first) |

## Architecture: Vendor as Context Layer

The vendor is a **configuration concept**, not a command concept. It changes *what data* you work with, not *how* you work. Build, serve, deploy, git — all work the same way, with different modules/branches/env vars provided by the context.

New code:
1. `jic vendor` command (create, checkout, sync, add, remove, list, status)
2. Config merge logic in the loader
3. Session/git extensions for root repo and branch naming

Existing commands (build, serve, deploy) receive the correct context and work without modification.

## 1. Config & File System Structure

### Type changes required

The following types must be extended:

- **`ProjectConfig`** (`core/types/config.ts`): add `type: "independent" | "submodules"` field (default: `"independent"`)
- **`DefaultsConfig`** (`core/types/config.ts`): add optional `env: Record<string, string>` field for environment variables
- **`JicState`** (`core/types/state.ts`): add `activeVendor?: string` field. `createEmptyState()` must also be updated.
- **`Session`** (`core/types/state.ts`): add `vendor?: string` and `rootBranch?: string` fields
- New Zod schema **`VendorConfigSchema`** for validating vendor config files

### Root config (`jic.config.json`)

New `project.type` field:

```json
{
  "project": {
    "name": "cicero",
    "type": "submodules"
  },
  "modules": { ... },
  "groups": { ... },
  "defaults": { ... }
}
```

- `type: "independent"` — current behavior (default if `project` is absent or `type` omitted, backward compatible)
- `type: "submodules"` — activates the vendor system

### Vendor config (`.jic/vendors/jic.config.<vendor>.json`)

The vendor config has its **own schema** (`VendorConfigSchema`), distinct from `JicConfig`. It does not redefine modules — it filters and overrides. Merge order: root config → vendor config (vendor wins).

```json
{
  "description": "Acme Corp customization",
  "modules": ["service-a", "frontend", "service-b"],
  "branches": {
    "master": "acme/master",
    "dev": "acme/dev",
    "build": "acme/build"
  },
  "nonVendorBranch": "master",
  "env": {
    "API_URL": "https://acme.example.com/api",
    "THEME": "acme"
  },
  "aws": { ... },
  "kubernetes": { ... }
}
```

- `modules`: **string array** — list of module names (not `Record<string, ModuleConfig>`). Acts as a filter on root config modules, and defines the implicit `@<vendor>` group.
- `branches`: **`VendorBranchConfig`** — a new type with keys `master`, `dev`, `build`, distinct from the existing `BranchConfig` (which uses `local`/`dev`/`main`). These are the branch names used for vendor checkout, sessions, and sync.
- `nonVendorBranch`: branch to checkout for modules NOT in this vendor (default: `"master"`)
- `env`: vendor-specific environment variables (merged into context, passed to build/serve/deploy)
- `aws`, `kubernetes`: deep-merge overrides on top of root config
- Rest: overrides that replace root config values

### The "root" vendor

Has its own config file (`.jic/vendors/jic.config.root.json`), generated by `jic vendor create root` or automatically at first setup:
- `modules`: all project modules
- `branches`: `{ master: "master", dev: "dev", build: "build" }`
- No env/aws/k8s overrides (unless customized by user)

### File system layout

```
project-root/
├── .git/                          ← root repo
├── .gitmodules
├── .jic/
│   └── vendors/
│       ├── jic.config.root.json
│       ├── jic.config.acme.json
│       └── jic.config.bigcorp.json
├── jic.config.json                ← root config with project.type: "submodules"
├── jic.state.json
├── service-a/                     ← submodule
├── service-b/                     ← submodule
└── frontend/                      ← submodule
```

### State (`jic.state.json`)

New `activeVendor` field:

```json
{
  "activeVendor": "acme",
  "sessions": { ... },
  "deployments": { ... }
}
```

`activeVendor` is always set when `type: "submodules"` — defaults to `"root"`.

## 2. Config Loader & Merge

### Loading flow

Current:
```
jic.config.json → jic.local.json → LoadedConfig
```

With vendor:
```
jic.config.json → jic.config.<vendor>.json → jic.local.json → LoadedConfig
```

The vendor config inserts **between** root config and local overrides. `jic.local.json` remains the last level for developer-specific overrides.

### Loader logic (`core/config/loader.ts`)

1. Load `jic.config.json` as today
2. If `project.type === "submodules"`:
   - Read `activeVendor` from `jic.state.json` (default: `"root"`)
   - Load `.jic/vendors/jic.config.<activeVendor>.json`
   - Merge: root config + vendor config (vendor wins)
3. Apply `jic.local.json` as today
4. Resolve modules

### Merge rules

- **`modules`**: the vendor config contains the **list** of active modules for that vendor. Does not redefine module structure (that stays in root config), but filters which modules are visible. `resolveModules([])` returns only these.
- **`branches`**: vendor override completely replaces `defaults.branches`
- **`defaults.env`**: deep merge (vendor env adds to/overrides root env)
- **`aws`, `kubernetes`**: deep merge (vendor can override endpoints, accounts, namespaces, etc.)
- **`groups`**: vendor automatically generates group `@<vendorName>` with its modules. Groups defined in root config remain available but operate only on modules in the active vendor (intersection).

### Module resolution with vendor

`resolveModules()` in `ExecutionContext` changes. **Note:** this is a behavioral change that only applies when `project.type === "submodules"`. Projects with `type: "independent"` (or no `project` field) are completely unaffected.

1. If empty args + active session → session modules (unchanged)
2. If empty args → modules of the active vendor (not all)
3. If args specified → resolve by name/alias/group, then **verify they are in the active vendor**, error otherwise

This single modification point propagates vendor behavior to all commands using `resolveModules()`.

## 3. Vendor Command

### Subcommands

| Command | Description |
|---------|-------------|
| `jic vendor list` | List available vendors (scan `.jic/vendors/`) |
| `jic vendor status` | Show active vendor, modules, current branches, divergence from master |
| `jic vendor create <name>` | Create config file + branches (master, dev, build) in root and submodules |
| `jic vendor checkout <name>` | Switch vendor: update state, checkout branches in modules |
| `jic vendor add <module>` | Add module to active vendor (create branches, update config) |
| `jic vendor remove <module>` | Remove module from vendor (update config, does not delete branches) |
| `jic vendor sync` | Merge from master into active vendor (root + submodules, fail-fast) |

### `jic vendor create <name>`

1. Generate `.jic/vendors/jic.config.<name>.json` copying structure from root config with renamed branches (`<name>/master`, `<name>/dev`, `<name>/build`)
2. In root repo: create branches `<name>/master`, `<name>/dev`, `<name>/build` **from `master`** (the base trunk, not from current branch)
3. In submodules listed in `modules`: same branch creation, same starting point (`master`)
4. Does not checkout — stays on current active vendor
5. Flag `--checkout`: after creation, switch to the new vendor
6. Flag `--from <branch>`: override the base branch (default: `master`)

### `jic vendor checkout <name>`

1. Verify no uncommitted changes (or `--force` to stash). Note: `--force` only overrides the uncommitted changes guard, NOT the active session guard (see Edge Cases).
2. Update `activeVendor` in `jic.state.json`
3. In root repo: checkout `<name>/master` (or branch specified in vendor config)
4. In vendor modules: checkout `<name>/master` branch
5. In non-vendor modules: checkout the branch specified by `nonVendorBranch` in vendor config (default: `master`)

Note: step 4-5 perform direct branch checkouts. `git submodule update` is NOT used here because the desired state is defined by vendor config branch names, not by root repo pointers. Root repo pointers are updated separately (see pointer update moments).

### `jic vendor sync`

1. Verify active vendor is not "root" (sync on root makes no sense)
2. In root repo: `git merge master` into `<vendor>/master`
3. For each vendor submodule (in order): `git merge master` into `<vendor>/master`
4. Fail-fast on first conflict
5. After manual conflict resolution, re-run command to continue

**Idempotency:** on re-run, `git merge master` is a no-op for modules where the merge was already completed (git merge is inherently idempotent — if `master` is already an ancestor of the current branch, the merge produces "Already up to date"). This ensures the command can be safely re-run to process remaining modules after conflict resolution.

6. On completion of all modules: update submodule pointers in root repo

### `jic vendor add <module>`

1. Verify module exists in root config
2. Create vendor branches in the submodule (`<vendor>/master`, `<vendor>/dev`, `<vendor>/build`)
3. Update `modules` list in vendor config file
4. Update submodule pointer in root repo

### `jic vendor remove <module>`

1. Verify module is in the vendor
2. Remove from `modules` list in vendor config file
3. Does not delete branches (user can do manually)

### `jic vendor status`

Example output:
```
Vendor: acme
Branch: acme/master
Modules: service-a, frontend (2/4)

  Module       Branch          Behind master
  service-a    acme/master     3 commits
  frontend     acme/master     1 commit

Non-vendor modules: service-b (master), shared-lib (master)
```

## 4. Vendor-Aware Sessions

### Session creation

`jic session start <name> [-m modules]`

Behavior when `type: "submodules"` with active vendor:

1. Base branch: `<vendor>/dev` (from active vendor config)
2. Branch naming: `<vendor>/feature/<name>` (e.g., `acme/feature/fixLogin`)
3. If `-m` specified: verify modules are a subset of active vendor, error otherwise
4. If `-m` not specified: use all vendor modules
5. Create branch in **root repo** + involved submodules
6. Update submodule pointers in root repo (specific workflow moment)

### Checkout, pause, resume

Work as today, with additions:
- Operate on root repo as well
- Checkout verifies coherence: if session was created under vendor `acme`, performs `jic vendor checkout acme` if needed before session checkout

### End session

`jic session end <name> [--merge]`

1. If `--merge`: merge `<vendor>/feature/<name>` → `<vendor>/dev` in root + submodules
2. Checkout `<vendor>/dev` in all session modules
3. Update submodule pointers in root repo
4. Update state

### Session state

Sessions track the vendor and root repo branch:

```json
{
  "sessions": {
    "fixLogin": {
      "vendor": "acme",
      "status": "active",
      "sessionBranch": "acme/feature/fixLogin",
      "baseBranch": "acme/dev",
      "rootBranch": "acme/feature/fixLogin",
      "rootBaseBranch": "acme/dev",
      "modules": {
        "service-a": {
          "branch": "acme/feature/fixLogin",
          "baseBranch": "acme/dev",
          "commit": "abc123"
        }
      }
    }
  }
}
```

The root repo is tracked via dedicated `rootBranch` and `rootBaseBranch` fields on the session, not as an entry in `modules` (since the root repo is not a module).

### Constraints

- Cannot start a session if active vendor does not match. Error with guidance to switch vendor.
- `jic session list` shows the vendor associated with each session.
- `jic session checkout <name>` automatically performs `jic vendor checkout <vendor>` if needed.

## 5. Git Operations with Root Repo

### Extended operations

When `type: "submodules"`, these commands operate on the root repo as well:

| Command | Root repo behavior |
|---------|-------------------|
| `jic git status` | Show root repo status (including modified submodule pointers) |
| `jic git commit` | Commit in submodules, then optionally `--update-root` for pointer commit |
| `jic git fetch` | Fetch in root repo + submodules |
| `jic git pull` | Pull in root repo + submodules |
| `jic git push` | Push in submodules, then push in root repo |
| `jic git log` | Include root repo commits in unified log |
| `jic git diff` | Include root repo diff |
| `jic git stash` | Stash in root repo + submodules |

### Operation order

For write operations (commit, push):
1. **Submodules first**, then root repo
2. Reason: root repo must point to updated submodule commits

For read operations (status, log, fetch):
1. **Root repo first**, then submodules
2. Reason: user sees general context before details

### `jic git commit --update-root`

1. Commit in specified submodules (or those with changes)
2. In root repo: stage modified submodule pointers (`git add <submodule-path>`)
3. Commit in root repo with auto-generated message: `"chore: update submodule pointers [service-a, frontend]"` (listing the updated submodules)
4. Without `--update-root`: commit only in submodules, root repo remains with "dirty" pointers

### Submodule pointer: automatic update moments

Pointers are updated only at these moments:

| Moment | Action |
|--------|--------|
| `jic vendor checkout <name>` | `git submodule update --recursive` |
| `jic vendor sync` | Commit pointers after sync completed |
| `jic vendor add <module>` | Commit pointer for new module |
| `jic session start` | Commit pointers after branch creation |
| `jic session end --merge` | Commit pointers after merge |
| `jic git commit --update-root` | Manual, on user request |

## 6. Impact on Build, Serve, Deploy

### Principle

These commands **are not modified in their internal logic**. Approach A ensures they receive from the context the correct modules and env vars for the active vendor.

### Build

- `jic build` without args → builds active vendor modules (not all)
- `jic build service-a` → works only if `service-a` is in active vendor, error otherwise
- `jic build @backend` → intersection with vendor modules
- Vendor env vars passed to build process through context

No modification to build code, pipeline, or dependency resolution. Filtering happens upstream in `resolveModules()`.

### Serve

- `jic serve` without args → serves active vendor modules
- Vendor env vars injected into spawned processes
- `jic serve --status` shows active vendor alongside running services

Minimal modification: pass vendor env vars to process. Rest unchanged.

### Deploy

- `jic deploy backend service-a` → uses AWS/K8s config from active vendor (if overridden in vendor config)
- If vendor has different AWS account, credentials and endpoints come from vendor config
- `jic deploy status` shows vendor alongside each deployment

Modification: `getAwsConfig()` and `getK8sConfig()` already read from context, which now includes vendor overrides. No change to deploy logic.

## 7. Error Handling & Edge Cases

### VendorError

New error type in the hierarchy:

```
JicError (exit 1)
├── ...existing...
├── KubernetesError (exit 10)
└── VendorError (exit 11)
```

Error cases:

| Situation | Error |
|-----------|-------|
| Module not in active vendor | `VendorError: Module "service-b" is not part of vendor "acme"` |
| Vendor checkout with uncommitted changes | `VendorError: Uncommitted changes in service-a. Use --force to stash.` |
| Vendor sync on "root" | `VendorError: Cannot sync vendor "root" — it is the base vendor.` |
| Vendor config file not found | `VendorError: Vendor config not found: .jic/vendors/jic.config.acme.json` |
| Session start with modules outside vendor | `VendorError: Modules "service-b" not in vendor "acme". Use "jic vendor add" first.` |
| Session/vendor mismatch | `VendorError: Session "fixLogin" belongs to vendor "acme", but active vendor is "bigcorp". Switch vendor first.` |

### Edge cases

**Vendor checkout during active session:**
Error: `Cannot switch vendor while session "fixLogin" is active. End or pause the session first.`
Note: the `--force` flag on `vendor checkout` only overrides the uncommitted changes guard. It does NOT bypass the active session guard — that always requires explicit session end/pause.

**Vendor sync during active session:**
Error: `Cannot sync vendor while session "fixLogin" is active. End or pause the session first.`
Reason: sync modifies base branches, doing it with an active session would create inconsistencies.

**Two vendors customizing the same module:**
Not a problem — each vendor has its own branches (`acme/master`, `bigcorp/master`). No conflict. The only risk is progressive divergence, mitigated by `jic vendor status` showing "behind master".

**Vendor deletion:**
Out of scope for now. User can delete config file and branches manually. Potential `jic vendor delete <name>` in the future.

**Vendor rename:**
Out of scope. Would imply renaming all branches in all repos. Too risky to automate.

## 8. Cross-Vendor Changes

When a change is needed by multiple vendors, the recommended workflow:

1. Work on `master` (switch to vendor "root")
2. Develop, test, merge to master
3. Sync interested vendors: `jic vendor checkout acme` → `jic vendor sync`
4. Repeat for each vendor

Principle: shared changes live on `master` (source of truth), vendor branches are deltas.

`jic vendor status` helps track divergence by showing how many commits each vendor is behind the base branch (as defined by the vendor's `branches.master` config).

## 9. `jic.local.json` Interaction

`jic.local.json` applies **after** vendor config and is vendor-agnostic — it applies uniformly regardless of active vendor. It is intended for developer-specific overrides (ports, paths), not vendor-specific ones.

If a developer needs different local overrides per vendor, they should use vendor config files for that. `jic.local.json` is for machine-specific settings that don't change with vendor context.
